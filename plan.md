# 数字货币自动化投资策略落地计划

制定一个数字货币自动化投资策略的落地计划，需要兼顾策略逻辑、技术实现、风险控制和持续迭代。下面是一份分阶段、可执行的实施路线图，可以根据自身技术背景和资金规模灵活调整。

---

## 一、总体原则（启动前必读）

- **生存第一**：数字货币波动极大，初期以保护本金为首要目标，切勿全仓运行。
- **小步快跑**：先用极小资金（或模拟盘）跑通全流程，再逐步放大。
- **拥抱不确定性**：市场结构会变，没有任何策略永久有效，必须建立持续监控与迭代机制。
- **安全红线**：API密钥权限最小化（关闭提币权限），使用独立的交易子账户，避免与长期持仓混用。

---

## 二、阶段划分与里程碑

### 阶段 0：基础设施搭建（第1-2周）

**目标**：拥有一个可运行策略代码、获取数据、并能通过API下单的骨架系统。

| 任务 | 说明 | 工具/建议 |
|------|------|----------|
| 选择交易所与API接入 | 注册并完成KYC，创建API Key（只开放交易和读取权限，严禁提币权限）。 | Binance、OKX、Bybit流动性好，API文档完善。推荐先选1家主流所。 |
| 搭建开发环境 | 选择编程语言，配置代码管理。 | 推荐 Python 3.9+ + Git。使用虚拟环境(venv/conda)隔离依赖。 |
| 数据获取模块 | 实现Websocket实时行情推送与REST补全历史数据。 | ccxt库统一接口，或交易所官方SDK。 |
| 下单执行模块 | 实现市价/限价单，查询余额、持仓、订单状态。 | 封装为一个独立的执行类，加全面日志。 |
| 本地日志与配置管理 | 将所有API调用、错误、订单事件记录到文件，配置文件集中管理。 | 使用python-decouple或yaml管理密钥，.gitignore中排除。 |

**阶段 0 检查点**：

- 能在终端实时打印BTC/USDT的最新价、买一卖一。
- 能通过脚本成功下一笔极小市价单（如0.001 BTC）并立即查询成交情况。
- 所有操作有详细日志，无密钥泄露风险。

---

### 阶段 1：策略原型与回测（第3-5周）

**目标**：定义清晰的交易逻辑，并通过历史数据回测初步验证有效性，避免"未来函数"和过拟合。

| 任务 | 说明 | 注意事项 |
|------|------|----------|
| 确定策略类型 | 选择一个逻辑简单、边界清晰的策略。新手切忌一开始就上机器学习。 | 建议从经典模型起步：双均线交叉、布林带回归、网格交易、动量突破。或者基于链上数据的宏观择时（如MVRV Z分数定投）。 |
| 用伪代码定义规则 | 明确开仓条件、平仓条件、仓位计算公式、止损止盈规则。 | 例如："当1小时线收盘价站上EMA50且RSI>50，用账户权益的2%开多；当跌破EMA50或单笔亏损达1%止损" |
| 回测框架搭建 | 选择或自研回测引擎，模拟滑点、手续费。 | 简单可用 backtrader、vectorbt；需要更精细控制可自建DataFrame遍历。手续费按现货0.1%、合约taker 0.05%设置。 |
| 历史数据清洗 | 确保数据覆盖至少一个牛熊周期（如2020至今），包含插针、宕机等异常情况。 | 注意交易所历史数据可能存在缺口，需对齐时间戳，处理极端插针（可设定异常波动过滤）。 |
| 回测与优化 | 跑出基础绩效：年化收益率、最大回撤、夏普比率、胜率、盈亏比。 | 避免过度优化：参数不超过2-3个，用样本外数据（最近30%时间）验证，观察权益曲线是否平滑。 |
| 敏感性与压力测试 | 改变参数、增加交易成本、随机延迟、模拟极端行情。 | 尤其要测试"5.19"或"3.12"这种级别暴跌时策略的亏损是否可控。 |

**阶段 1 检查点**：

- 策略逻辑能用一句话清晰描述给非交易者听懂。
- 回测结果在样本外数据上未出现显著恶化。
- 最大回撤在自己可接受的心理范围内（建议初期控制在10-20%以内）。

---

### 阶段 2：模拟盘/极小资金实盘（第6-8周）

**目标**：让系统在真实市场环境中无人值守运行，暴露策略逻辑以外的工程问题。

| 任务 | 说明 |
|------|------|
| 搭建实盘运行骨架 | 用异步框架或主循环实现：接收行情 -> 判断信号 -> 执行订单 -> 记录状态 -> 异常处理。 |
| 模拟/小资金运行 | 先用交易所的模拟盘（如有）或创建独立子账户，存入极小资金（如总仓位的1-5%），全自动运行1-2周。 |
| 监控与告警 | 至少监控：程序存活、网络断连、订单被拒、持仓异常、连续亏损。通过邮件/Telegram/微信发送告警。 |
| 记录所有偏差 | 对比实盘成交与理论信号的滑点、延迟、订单失败率，评估真实冲击成本。 |

**阶段 2 检查点**：

- 系统能稳定无人值守运行72小时以上，无崩溃、无异常挂单。
- 实盘与回测的成交价偏差在可接受范围。
- 告警通道畅通，收到告警后能远程干预。

---

### 阶段 3：渐进式放量与风控强化（第9周起）

**目标**：在严格风控框架下逐步增加资金，并建立运营手册。

| 任务 | 说明 |
|------|------|
| 建立每日检查清单 | 记录账户权益、持仓、未成交订单、资金费率、系统日志异常。 |
| 设定硬风控指标 | 例如：当日累计亏损超总资金3%时强制停止交易并平仓；单币种最大持仓不超过20%；连续止损N次后暂停等。 |
| 资金管理规则 | 初期可固定比例（如每次开仓2%权益）；熟练后可引入凯利公式或波动率调整。**严禁马丁格尔式逆势加仓**。 |
| 部署到服务器 | 迁移至云服务器（AWS/阿里云/低延迟VPS），设置开机自启和守护进程，保证7x24运行。 |
| 版本管理与回滚计划 | 每次策略/代码变更必须经过回测验证，并保留上一个稳定版本，以便快速回滚。 |
| 多策略组合 | 不同策略（如趋势+网格）分配不同仓位，降低单一策略失效风险。 |
| 定期复盘机制 | 每周/每月生成绩效报告，分析盈亏来源，识别策略退化信号。 |
| 灾备方案 | 准备手动干预脚本（一键平仓、暂停所有策略），异地备份配置与日志。 |

**阶段 3 检查点**：

- 资金曲线运行1个月，最大回撤未超过预警线的70%。
- 所有风控规则已通过代码强制执行，而非人工依赖。
- 有完整的运维文档，出现问题时有清晰的应急预案。

---

### 阶段 4：持续迭代与进阶（长期）

**目标**：根据市场变化持续优化，探索更高级策略。

| 任务 | 说明 |
|------|------|
| 策略生命周期管理 | 定期（每季度）评估策略是否失效，设置淘汰标准（如连续3个月夏普比率<0）。 |
| 新策略研发管道 | 保持2-3个备选策略在回测阶段，随时准备替换失效策略。 |
| 跨市场/跨品种扩展 | 在验证有效后，可扩展到其他交易所、其他币种，分散流动性风险。 |
| 高级方向探索 | 可考虑：机器学习特征工程、链上数据融合、跨交易所套利、期权波动率策略。 |

## 四、风险提示

1. **策略失效风险**：市场结构变化可能导致策略完全失效。
2. **技术风险**：网络中断、API限制、交易所故障可能导致无法及时止损。
3. **操作风险**：配置错误、代码bug可能导致意外大额亏损。
4. **监管风险**：部分地区禁止加密货币交易，需关注合规要求。

## 七、关键决策：现货 or 合约？

- **建议初期只做现货**：省去资金费率、强制平仓等复杂性，压力更小。
- **如果要做合约**：严格限定低倍杠杆（如2-3x），且必须从策略层面就计算好爆仓价，并在代码中设置远超交易所标记价的预警平仓线。

## 九、常见陷阱与对策

| 陷阱 | 对策 |
|------|------|
| 回测收益惊人，实盘稳定亏损 | 检查是否用了未来数据、忽略手续费/滑点、过拟合。强制保留30%样本外数据验证。 |
| 系统半夜崩溃，错过行情或无法止损 | 加入try-except全捕获，一旦异常立即发送告警并尝试安全退出（取消所有挂单）。 |
| 交易频率超出交易所限制 | 在代码中加入频率计数器，接近限频时自动降速或切换只读。 |
| API返回非预期数据导致逻辑错乱 | 对所有外部输入做校验，如价格<=0、数量超限、余额不足等，抛出明确异常。 |
| 心理上受不了连续亏损，手动干预 | 策略上线后，手动干预等于毁灭纪律。如果无法信任系统，说明回测和风险场景测试还不够。返回去重做阶段1。 |

## 十、初期最小可行策略示例

### "趋势跟随 + 波动率定投"现货策略骨架

**策略逻辑**：

1. **判断趋势**：BTC周线收盘价 > EMA30，视为上升趋势，只做多；否则空仓或持有稳定币。
2. **开仓信号**：在上升趋势中，每日定时（如UTC 0点）检查，若当日凌晨到当前时间价格比前一日收盘跌超2%，则用账户现金的5%买入现货。
3. **止盈**：单币种盈利超30%卖出一半，剩余移动止盈（如从最高点回落10%全平）。
4. **止损**：单币种亏损15%无条件卖出。

**伪代码示例**：

```python
def check_trend(btc_weekly_close, ema30):
    return btc_weekly_close > ema30

def check_entry_signal(current_price, prev_close, drop_threshold=0.02):
    return (prev_close - current_price) / prev_close > drop_threshold

def calculate_position_size(account_balance, position_pct=0.05):
    return account_balance * position_pct

def execute_strategy():
    if check_trend(btc_weekly_close, ema30):
        if check_entry_signal(current_price, prev_close):
            position_size = calculate_position_size(account_balance)
            place_order('buy', position_size)
```

**策略特点**：
- 逻辑极简，回测容易
- 不容易过拟合
- 适合作为第一个自动化项目练手
- 现货无爆仓风险
